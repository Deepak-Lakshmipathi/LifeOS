Get started & test

The whole slice backbone (S1–S19) is shipped and the dashboard is live on the internet — PWA, vault sync, and the Telegram bot with all four capture modalities. This page is the user-testing guide: open (or install) the live app, then walk the checklist to verify the flow end–to–end. Local dev instructions follow for making changes.

0 · It’s live — just open it

The dashboard is hosted on GitHub Pages and auto-deploys on every push to master:

deepak-lakshmipathi.github.io/LifeOS

  1. Open it on desktop or phone. On first load it prompts once for your vault access token (a GitHub fine-grained PAT for the vault repo, Contents: Read + Write). It’s stored only in that browser — never in the app bundle.
  2. It clones your private vault (LiveOS-VaultRepo) in-browser through a self-hosted Cloudflare Worker proxy, then loads your tasks. Add or complete one and it commits + pushes straight back to the vault.
  3. Install it — use the browser’s “Install app” (address-bar icon or ⋮ menu) for a standalone, offline, full-screen app.

Wrong token or a load error? The app shows a panel with the reason and a Re-enter token button (clears the stored PAT and reloads). The PAT + all git traffic transit only your own Cloudflare account — not any third party.

1 · Run the dashboard locally (for changes)

The dashboard is a Vite + React + TypeScript PWA. From the repo root:

npm install
npm run dev            # dev server, hot reload
npm run build && npm run preview   # production build

Open the URL Vite prints (usually http://localhost:5173). On first run with an empty database it seeds 107 tasks from seed_tasks_detailed.json (idempotent; add ?noseed to skip).

Vault-backed mode

By default the app is local-only (IndexedDB). To make the Obsidian vault the real source of truth (git-as-transport read + write), copy .env.example → .env (gitignored), fill in VITE_VAULT_REPO_URL, VITE_VAULT_PAT (Contents: Read + Write) and the VITE_VAULT_CORS_PROXY URL, then:

VITE_VAULT=1 npm run dev

Writes commit locally (offline-safe) and best-effort push. The hosted build gets the same config from CI env + the VAULT_CORS_PROXY repo variable — the PAT is entered at runtime and never baked in. The CORS proxy is the self-hosted Worker in cors-proxy/ (see its README to deploy).

2 · Run the Telegram bot

A standalone Node worker in services/bot/ long-polls Telegram and writes to the same vault over its own git credential. Copy .env.example → .env (gitignored) and fill in all six secrets:

TELEGRAM_BOT_TOKEN         # from @BotFather
BOT_VAULT_PAT              # vault-repo PAT (Contents: R+W), separate from the PWA's
BOT_VAULT_REPO_URL         # same vault repo the PWA points at
ANTHROPIC_API_KEY          # Claude — NLU + photo vision
GROQ_API_KEY               # Whisper — voice transcription
OWNER_TELEGRAM_CHAT_ID     # your chat id (via @userinfobot); every other id is ignored
cd services/bot && npm install && npm start

The bot serves a single owner. Any chat id that isn’t OWNER_TELEGRAM_CHAT_ID is a complete no-op — no reply, no Claude call, no vault write.

3 · Dashboard walkthrough

A glass command center with a bottom tab bar: Now · Domains · Pulse · +.

  1. Capture a task — tap +. One field parses shorthand in any order: #domain (fuzzy-matched to one of 7; no match → Inbox), !1/!2/!3 priority, when … or ~ … for a “done when” finish line, /project, and the rest is the title. A live preview shows the parse before you commit. e.g. Book dentist #body !2 when confirmed /health-admin
  2. Work the NOW queue — the Now tab is the balance brain: priority-ranked, capped ~2 per domain, injecting one ❄ rescue task from your coldest domain. Top 3 live; the rest fold under “Up next / Later”.
  3. Complete a task — tap the ● dot. Ring pulse, fade-and-fold, a 3-second Undo toast. (Haptic buzz on mobile.)
  4. Read the map — the Domains tab shows a warmth tile per domain: glow + a one-word state, hot → cold.
  5. See trends — the Pulse tab: done-this-week count, a 7-day sparkline, and warmth standings.
  6. Install it — use the browser’s “Install app”; it runs offline, full-screen, like native.

4 · Bot walkthrough

Message the bot from your owner account. It confirms every create, and gates every edit and delete behind an explicit reply.

  1. Text → create — send a free-form task (call the CA about GST, life admin, high priority). Claude classifies + extracts, writes a task line to the vault (real commit, non-empty id::), and replies ✓ added ….
  2. Photo → vision — send a photo (whiteboard, sticky notes, a list). Claude vision extracts the tasks (cap 20) and replies with a numbered batch. Reply all, none, or a subset (1,3) to choose which get created.
  3. Voice → transcribe — send a voice note. Groq Whisper transcribes it; a confident transcript flows through the same create pipeline with the reply echoing heard: ‘…’ → ✓ added. A muffled/empty one short-circuits to “mind typing it instead?” — no write.
  4. Edit / delete → confirm — ask to change or remove a task (mark call the CA done, delete the dentist task). The bot fuzzy-matches the target, echoes the exact change, and waits: it only commits on a y. Ambiguous reference → a numbered candidate list; n or a 2-minute timeout cancels.

5 · Testing checklist

Dashboard (PWA)

  • App loads and seeds tasks on first run (Now tab shows a ranked queue).
  • Capture parses shorthand — type the example above, confirm the preview splits domain / priority / done-when / project.
  • Complete a task → ring pulse plays, Undo toast appears, Undo restores it.
  • Domains tab shows 7 tiles; completing a task warms its domain.
  • Pulse count and sparkline update after completions.
  • Reload the page — tasks survive (local persistence).
  • Go offline (DevTools → Network → Offline) — the app still opens and works.
  • Install as a PWA and launch it standalone.
  • Vault mode: add a task, then confirm it lands as a commit in the vault repo.

Telegram bot

These exercise the live git-network + Telegram + Claude/Groq paths that CI can’t reach — the owner-only sign-off before trusting the bot in production (full script: afk-pipeline-out/s16c-verify-checklist.md).

  • Text create → a real commit lands (author LifeOS Bot, non-empty id::), pushes to the remote, and shows on the PWA (VITE_VAULT=1) after refresh; bot replies ✓ added.
  • Offline resilience — cut the bot’s network, text a task: it still replies ✓ added (local commit); reconnect and the commit reaches the remote (not lost).
  • Non-owner ignored — message from a different account: no reply, no commit, no Claude call.
  • Ambiguous domain → Inbox — a task with no confident domain lands under Inbox/, and the reply shows Inbox.
  • Photo batch — a photo yields a numbered list; all creates every task, none cancels, 1,3 creates only that subset.
  • Voice — a clear voice note transcribes and creates (reply echoes heard: …); a muffled/empty one gets the retype prompt, no write.
  • Confirm-gated edit/delete — update/delete waits for y; n or timeout cancels with no write; a stale target (changed since asked) is rejected, not mis-applied.
  • No secret (PAT, bot token, API keys) appears in the bot’s logs; the local .vault-clone/ is never committed to this repo.
